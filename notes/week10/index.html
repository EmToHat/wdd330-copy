Validating Forms

Client-side Form Validation
Before submitting data to the server, ensure that all required form controls are properly filled out in the correct format.

This form helps to ensure that the data submitted will match the requirements made by the various form controls.

Client-side Validation
Is an initial check and is an important feature of good UI (user experience). It catches invalid data on the client-side, which the user can fix straight away.

If the data gets to the server-side, it will be rejected and cause a delay going on a round trip to the server and back to the client side.

Client-side validation shouldn't be considered as an exhaustive security measure. My Apps should ALWAYS perform security checks on any form-submitted data both on server-side and client-side.

Client-side validation is easy to bypass, so it is easy for malicious users to send bad data to the server.

Form Validation, what is it?
e.g. Registration Form, the user will recieve feedback when data was entered incorrectly. Messages could be: "This field is required", "Please enter your phone number in the format xxx-xxxx", "Please enter a valid email address", "Your password needs to be between 8 and 30 characters long and contain one uppercase letter, one symbol, and a number."

These are called form validation. When the user enters data, the browser/or web server will check to see if the entered data is correct and will let the user know, especially if the data is incorrect.

Client-side Validation: It is validation done in the browser.

Server-side Validation: It is validation done on the server.

Forms
As web developers we want to make filling out web forms as easy as possible for our users. So why is do we insist on validating those forms?

There are three MAIN reasons:
1. We want to get the right data, in the right format.
2. We want to protect our users' data.
3.We want to protect ourselves.

NEVER TRUST data passed to the server from the client. This includes if the form is validating correctly, a malicious user can stil alter the network request.

Client-Side Types
Client-side has two types: Built-in form validation, and JavaScript.

Built-in form validation: uses HTML5 form validation features, this type of validation doesn't usually require too much JS. The built-in form validation has better performance than JS, but it isn't as customizable.

JS form validation: is coded using JS. It is completely customizable, but you need to create all the validation or use a library.

Built-in form Validation
required, minlength, maxlength, min, max, type, pattern

Using Fetch

The Fetch API allows a JS interface to access and manipulate parts of the HTTP pipline like requests and responses.

The fetch specification differs from jQuery.ajax() in the following significant ways:
The Promise returned from fetch() won't reject on HTTP error status even if the response is an HTTP 404 or 500.

Basic Fetch Request
fetch('http://example.com/movies.json')
.then(response => response.json())
.then(data => console.log(data));

The fetch() method can optionally accept a second parameter, an init object that allows you to control a number of different settings.
